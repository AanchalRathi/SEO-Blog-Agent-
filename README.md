## Overview

Most SEO content tools either require manual keyword research or produce generic blog posts disconnected from a brand's actual voice. This system solves both:

1. **Real keyword discovery** — pulls live data from Google Autocomplete and Serper.dev, no static keyword lists
2. **Intent-based scoring** — ranks keywords by transactional/informational/commercial intent, brand relevance, and long-tail value
3. **Brand-aware generation** — RAG retrieval pulls real brand context (pricing, features, USPs) from uploaded documents so blogs reflect the actual company, not generic AI output
4. **Adaptive content format** — automatically selects from 6 blog structures based on keyword intent (listicle, comparison, educational, FAQ, case study, standard)
5. **Persistent blog management** — every generated blog saved to PostgreSQL with status tracking (draft/published/archived)

## Pipeline Walkthrough

1. **User input** — company name, niche, target audience, competitors, tone, region, and optional brand documents uploaded via the Streamlit sidebar

2. **Keyword Discovery** (`keyword_discovery.py`)
   - Builds search seeds dynamically from company context — brand seeds, niche seeds, competitor comparison seeds, and informational seeds
   - Queries Google Autocomplete (free, no key needed) + Serper.dev for related searches and People Also Ask results

3. **Keyword Scoring** (`keyword_analysis.py`)
   - Classifies intent: transactional / informational / commercial
   - Scores each keyword by intent strength, brand term relevance, long-tail value, and recency signals
   - Filters off-brand keywords (job listings, salary queries, financial/investor news) that would produce hallucinated content
   - If the user typed a specific query, it takes priority over auto-discovered keywords regardless of score

4. **Blog Type Selection** (`crew.py`)
   - Rule-based selection from 6 formats based on keyword signals:
     - `comparison` — vs / versus / better / alternative keywords
     - `educational` — how to / guide / tips / steps keywords
     - `listicle` — best / top / ranking keywords
     - `faq` — question-based keywords
     - `case_study` — success story / result keywords
     - `standard` — general transactional or commercial keywords

5. **RAG Brand Context** (`tools/rag_tool.py`)
   - Loads uploaded brand documents (PDF / TXT / MD)
   - Chunks documents and embeds via Cohere API into ChromaDB
   - Each company gets its own isolated ChromaDB collection — no cross-contamination between brands
   - Retrieves the most semantically relevant chunks for the target keyword

6. **Blog Generation** (`blog_generator.py`)
   - Single Groq API call with structured prompt template for the selected blog type
   - Brand context injected directly into the prompt so the LLM uses real pricing, features, and USPs instead of hallucinating
   - Outputs SEO title, meta description, URL slug, and 900-1100 word blog post

7. **Persistence** — saved to PostgreSQL, viewable and manageable in the Blog History tab

## Setup — Run Locally

**Prerequisites:** Python 3.11, Docker (optional), Groq API key, Serper API key, Cohere API key, PostgreSQL connection string

```bash
# Clone the repo
git clone https://github.com/AanchalRathi/SEO-Blog-Agent-
cd seo-blog-agent

# Set up environment variables
cp .env.example .env
# Fill in .env with:
# GROQ_API_KEY=
# SERPER_API_KEY=
# COHERE_API_KEY=
# DATABASE_URL=
```

**Option 1 — Run with Docker (recommended):**
```bash
# Build the image
docker build -t seo-agent .

# Run the API
docker run -p 8000:8000 --env-file .env seo-agent

# Run the UI (separate terminal)
docker run -p 8501:8501 --env-file .env \
  -e API_URL=http://localhost:8000 \
  seo-agent \
  streamlit run app.py --server.port 8501 --server.address 0.0.0.0
```

**Or use Docker Compose for both services at once:**
```bash
docker-compose up
```

**Option 2 — Run without Docker:**
```bash
# Create and activate virtual environment
python -m venv venv
venv\Scripts\activate        # Windows
source venv/bin/activate     # Mac/Linux

# Install dependencies
pip install -r requirements.txt

# Terminal 1 — API
uvicorn api:app --reload --port 8000

# Terminal 2 — UI
streamlit run app.py
```

Visit `http://localhost:8501`

## Known Limitations & Future Improvements

**Current constraints:**
- Brand documents and ChromaDB vectorstore are cleared on Render service restart due to ephemeral filesystem — re-upload required after cold starts. Persistent storage via Cloudflare R2 is the planned fix.
- Render free tier spins down after 15 minutes of inactivity — first request after idle takes 30-50 seconds to wake up. Expected behavior on free tier.
- Groq free tier TPM limits prevent multi-agent mode in production — CrewAI implementation is complete and preserved in `crew.py`, ready to enable on Groq Dev tier.

**Planned improvements:**
- Persistent document storage via Cloudflare R2
- CrewAI multi-agent mode on Groq Dev tier
- User-selectable blog type override in UI
- Batch blog generation for content calendars
- Blog quality scoring using LLM-as-judge pattern

## Engineering Notes

This project involved solving several real production challenges that don't appear in tutorials:

**Async job pattern for long-running AI pipelines**
Render's 30-second request timeout would kill blog generation mid-run. Solved by returning a `job_id` instantly from POST /generate and having the UI poll GET /jobs/{id} every 5 seconds. Each individual HTTP request stays under 1 second regardless of how long the pipeline takes.

**Memory constraints on free tier**
Local HuggingFace sentence-transformer models (~60-90MB on disk, 200-300MB in RAM during load) caused consistent silent OOM kills on Render's 512MB free tier. Diagnosed via instance failure events in Render's event timeline. Solved by switching to Cohere's embedding API — embeddings now happen via HTTP call with zero in-process model loading.

**Background task execution on Render**
Python's `threading.Thread(daemon=True)` was silently killed the moment the request returned on Render's worker process. Replaced with FastAPI's native `BackgroundTasks` which runs after the response is sent within the same async event loop — the correct approach for this pattern.

**Brand doc cross-contamination**
Streamlit's file uploader persists uploaded files in widget state across reruns within the same session. Switching companies without clearing the uploader caused multiple companies' brand docs to merge into the same ChromaDB collection. Solved by scoping the uploader widget key to the company name (`key=f"uploader_{company_name}"`), which resets the widget whenever the company changes.

**Keyword quality and content relevance**
Pure transactional keywords (coupon codes, promo codes) always scored highest due to strong intent signals, producing repetitive low-value content. Solved with score penalties for coupon-hunting keywords, off-brand topic filtering (job listings, salary queries, financial news), and a user query priority override that respects what the user actually typed regardless of auto-discovered keyword scores.

**Neon PostgreSQL connection drops**
Long polling sessions (3+ minutes) caused Neon's serverless Postgres to drop idle connections mid-poll, producing SSL EOF errors. Solved with `pool_recycle=280` on the SQLAlchemy engine, forcing connection refresh before Neon's timeout window.
